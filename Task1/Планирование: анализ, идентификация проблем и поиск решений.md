# 1. Проблемные места

## Cуществующие

- Клиенты жалуются, что над их изделием работают долго и проблемы не на стороне производства. (Вероятно на стороне системы).
- После подключения продавцов ювелирных изделей по API к MES - жалобы на просроченные заказы увеличились.
- Пользователи API (представители компаний) жалуются, что не получают заказы.
- Операторы часто жалуются, что система долго прогружается, когда заходят на первую страницу MES. (На первой странице находится список заказов со статусами).

- Ручные деплои в release и продакшен окружения -> низкий TTM (time-to-market).

## Потенциальные

Повышенный bus фактор:

- 1 разработчик на C#
- только 1 QA
- нет полноценного специалиста на React

Это может быть проблемой в случае если сотрудник, например, уйдет в отпуск или уволится.

Дубликация схожими технологиями по техническому стеку, может быть потенциальной проблемой для найма, взаимозаменяемости, а также при переиспользовании технических решений:

- React и Vue
- Java и C#

А также:

- Не охвачена пирамида тестирования: отсутствует множество видов тестирования, например, QA реализовал несколько автотестов - можно развивать эту инициативу.
- Нет трейсинга, который мог бы отследить статус заказа от создания и до выполнения.
- Приложение не масштабировано, а также в нем плохая наблюдаемость (observability)

## Инициативы в порядке приоритета

- Добавить мониторинг, трейсинг и логирование (так как с помощью него можно будет оценить дальнейшие инициативы)
- Добавить кеширование на Redis (первым пунктом, потому что это не сложно, но уже будет пример роста (fast-growth))
- Найм новых сотрудников для перехода на микросервисы и снижение bus factor
- Добавить инстансов приложениям
- Улучшить тестирование, добавить автоматизированное тестирование
- Автоматизировать CI/CD, автоматические релизы в rc и prod окружения с возможностью rollback
- Шардировать БД, а также реплецировать. Исторические данные можно разместить на HDD, а более свежие на SSD
- Перейти на микросервисы
- Перейти с Rabbit на Kafka, чтобы сохранять очередь, а также была возможность более простого горизонтального масштабирования

Если бы был выбор из только 3ех инициатив, то я бы выбрал:

- Автоматизация CI/CD
- Переход на микросервисы
- Мониторинг + трейсинг + логирования

Это был бы минимальный маст-хев, который позволил бы в этом проекте спокойнее спать. Автоматизация CI/CD увеличит TTM, снизит возможность совершить ошибку. Микросервисы могли бы помочь распределить нагрузку и вертикально масштабировать определенные сервисы, которые страдают. Мониторинг, трейсинг и логирование позволят команде разработке лучше понимать, какие места в проекте нужно оптимизировать с точки зрения разработки и давали бы более полное представление о том, что происходит в проекте.

## Архитектура через полгода

- Добавлено кеширование на Redis
- Добавлены мониторинг, трейсинг и логирование
- У каждого приложения (сервиса) есть несколько инстансов, которые могут снизить нагрузку
- Тестирование частично автоматизировано, команде QA не нужно делать регрес после каждого релиза
- CI/CD автоматизировано, релизы на rc и prod полностью автоматические без ручного труда
- БД шардировано и есть репликация
- MES при помощи Strangler Fig начинает постепенно заменяться на микросервисы
- Rabbit выведен, введена Kafka

Но это очень и очень позитивный сценарий :)
