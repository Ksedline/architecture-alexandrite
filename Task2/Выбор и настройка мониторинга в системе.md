# 2. Мониторинг

## Мотивация

- Быстрое обнаружение и устранение проблем

Мониторинг позволяет выявлять сбои или отклонения от нормы в режиме реального времени и быстро принимать меры, чтобы минимизировать простой и финансовые потери.

- Оптимизация производительности

С помощью мониторинга можно анализировать загрузку оборудования, использование ресурсов (процессор, память, сеть) и находить «узкие места», которые требуют оптимизации.

- Повышение безопасности

Мониторинг помогает выявлять подозрительную активность, попытки несанкционированного доступа и другие угрозы безопасности, что позволяет своевременно реагировать на инциденты.

- Прогнозирование и планирование

Анализ исторических данных мониторинга помогает прогнозировать будущие нагрузки, износ оборудования и планировать необходимые ресурсы или обновления.

- Снижение расходов

Мониторинг позволяет выявлять неиспользуемые ресурсы, оптимизировать их использование и, как следствие, сокращать затраты на обслуживание.

- Соблюдение нормативных требований и SLA

Для компаний, предоставляющих услуги, мониторинг помогает контролировать выполнение соглашений об уровне сервиса (SLA) и соответствовать требованиям регуляторов.

- Улучшение клиентского опыта

Стабильная работа сервисов напрямую влияет на удовлетворенность пользователей. Мониторинг позволяет поддерживать высокий уровень доступности и производительности.

Плюс — можно собирать бизнес-метрики (например, частоту использования определённых функций), что даёт понимание востребованности разных частей системы. Это помогает развивать продукт осознанно, опираясь на реальные данные.

## Выбор подхода к мониторингу

Для взаимодействия с пользователями подходит RED, чтобы акцентировать внимание на частоту запросов и ошибок, а еще следить за метрикой скорости выполнения

Для взаимодействия с API и Rabbit - 4 Golden Google Signals и USE. Здесь акцент будет на то, как сервера справляются с повышенной нагрузкой

## Что отслеживать

Основные метрики, которые стоит отслеживать:

1. Number of dead-letter-exchange letters in RabbitMQ
Количество сообщений, перенаправленных в dead-letter exchange. Важно отслеживать, чтобы понять, сколько заказов "потерялось" и вовремя принять меры для исправления проблем.

2. Number of message in flight in RabbitMQ
Количество сообщений, которые находятся в процессе доставки и ещё не подтверждены потребителем. Если метрика растёт, значит система не успевает обрабатывать сообщения, что может привести к задержкам и потере данных. 

3. Number of requests (RPS) for internet shop API
Частота запросов к API интернет-магазина - отражает нагрузку на интернет-магазин. Мониторинг помогает выявлять пиковые нагрузки и планировать ресурсы.

4. Number of requests (RPS) for CRM API
Аналогично предыдущему, эта метрика показывает нагрузку на CRM-систему. Позволяет контролировать пропускную способность и своевременно масштабировать сервис.

5. Number of requests (RPS) for MES API
Отслеживает количество запросов к MES. Важно для понимания текущей нагрузки и предотвращения перегрузок, что обеспечивает своевременную обработку заявок.

6. CPU % for MES API
Процент использования процессора сервисом MES. Высокая загрузка CPU может привести к деградации производительности, поэтому важно следить, чтобы ресурсов хватало и вовремя масштабировать или оптимизировать сервис.

7. Memory Utilisation for MES API
Процент использования оперативной памяти MES API. Недостаток памяти приводит к замедлению работы и возможным сбоям, поэтому мониторинг помогает поддерживать стабильность и производительность.

8. Response time (latency) for shop API
Время отклика интернет-магазина на запросы. Эта метрика напрямую влияет на удовлетворённость пользователей - чем меньше задержка, тем лучше опыт взаимодействия с сервисом.

9. Response time (latency) for CRM API
Аналогично, показывает скорость обработки запросов CRM. Высокая задержка может указывать на проблемы в системе и требует внимания.

10. Response time (latency) for MES API
Время отклика MES API. Важная метрика для оценки производительности и своевременной обработки заявок.

11. Number of HTTP 500 for shop API
Количество ошибок сервера (HTTP 500) для интернет-магазина. Ошибки снижают качество сервиса и пользовательский опыт, поэтому нужно оперативно реагировать на рост этой метрики.

12. Number of HTTP 500 for CRM API
Аналогично, количество ошибок сервера в CRM API. Помогает выявлять проблемы и предотвращать сбои.

13. Number of HTTP 500 for MES API
Ошибки сервера в MES API. Мониторинг позволяет быстро обнаружить и устранить причины сбоев, обеспечивая стабильную работу системы.

14. Number of simultanious sessions for MES API
Поможет отследить, сколько в текущий момент происходит расчетов. Если их мало или они отсутствуют при большом потоке заказов, то это признак, при котором возможно стоит принять меры.
